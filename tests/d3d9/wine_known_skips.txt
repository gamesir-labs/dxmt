# Test functions skipped up front, one per line as "<module> <function>", each
# with a comment saying why. Unlike the crash-discovered skips the suite finds at
# run time, these are declared: functions that are non-deterministic or hang for
# a reason outside DXMT, so gating on them would make the suite flaky. A declared
# skip is seeded into every run, counted toward coverage, and is not a failure.
# Remove an entry once the underlying cause is fixed.

# d3d9ex: these three call ChangeDisplaySettings themselves and then assert the
# system screen size changed. winemac.drv applies a mode change only while the
# process is the active Cocoa app and only once the deactivation settles, so the
# readback is non-deterministic run to run (the screen-size assertions pass or
# fail by timing). DXMT is not in that path. test_wndproc additionally trips a
# winemac.drv deadlock on the fullscreen focus-loss minimize:
# https://github.com/gamesir-labs/wine/issues/2
d3d9ex test_wndproc
d3d9ex test_reset
d3d9ex test_reset_ex

# device: same two classes as above. test_wndproc trips the winemac.drv deadlock
# (https://github.com/gamesir-labs/wine/issues/2) and test_reset changes the
# display mode, both of which hang the run. test_render_target_device_mismatch
# also hangs here, in winemac window teardown across two devices; treat it as the
# same platform hang until proven otherwise.
device test_wndproc
device test_reset
device test_render_target_device_mismatch

# device: these four are non-deterministic on the headless CI runner. They call
# SetForegroundWindow, which intermittently fails with no real desktop focus, and
# assert window rects and device-lost state that depend on that focus, so their
# failing line set varies run to run (measured: two runs differed only in these
# functions). Skipped so the gate stays deterministic; the rest of the module
# is stable and baselined (the x87 control-word check is skipped below).
device test_wndproc_windowed
device test_window_style
device test_lost_device
device test_window_position

# device: test_fpu_setup verifies the x87 FPU control word round-trips: it does
# set_fpu_cw(0xf60) then get_fpu_cw() BEFORE any device call and asserts the read
# back matches. Rosetta's x87 emulation does not preserve the control word (the
# read returns 0x7f regardless), so the check cannot pass on this host, entirely
# independent of DXMT (which reprograms the FPU correctly at CreateDevice via
# setupFpu). An environment limit, not a DXMT gap.
device test_fpu_setup
