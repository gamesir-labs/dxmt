# D3D9 Shader Oracle

This directory vendors vkd3d's `shader_runner` as an in-tree numeric oracle for
the d3d9 frontend. Each `corpus/*.shader_test` draws a quad through the real
`IDirect3D9` API, reads the render target back and probes pixels with per-probe
ulps tolerance, which is the numeric tier the airconv DXSO->AIR goldens cannot
provide.

The sources come from upstream vkd3d:

```
https://gitlab.winehq.org/wine/vkd3d.git
commit 5fc3d465dbbbc51c4b57bfa48286f1401754c9c8 (vkd3d-1.19-160-g5fc3d465)
```

## What is vendored

Only the pieces the d3d9 oracle needs, kept as close to upstream as possible:

- `runner/shader_runner.c`, `runner/shader_runner.h`, `runner/utils.h`: the
  shared, backend-independent test framework (parser, probe/ulps comparison,
  quad draw dispatch), byte-identical to upstream `tests/`.
- `runner/shader_runner_d3d9.c`: the d3d9 executor, one of upstream's five
  backends (the d3d11/d3d12/gl/vulkan backends are not vendored).
- `runner/include/`, `runner/include/private/`: the vkd3d support headers the
  framework includes (`vkd3d_windows.h`, `vkd3d_shader.h`, `vkd3d_test.h`,
  `vkd3d_common.h`, and friends).
- `runner/idl/`: the `dxcompiler.idl` widl generates the `IDxcCompiler3`
  interface from, the same one `tests/vkd3d-proton` uses. The d3d9 executor
  only `dlopen()`s `dxcompiler.dll` if present; SM6 is out of d3d9 range, so it
  is never instantiated here.
- `runner/local/`: the small DXMT-specific build shims:
  - `config.h`: empty. In upstream this is generated by `configure.ac` to
    advertise optional host features; the d3d9-only cross build needs none.
  - `vkd3d_d3dcommon.h`, `vkd3d_d3d12.h`, `vkd3d_dxgiformat.h`: redirects to the
    mingw-w64 SDK's `d3dcommon.h` / `d3d12.h` / `dxgiformat.h`, so the widl-
    generated vkd3d variants and their dependency chains are not dragged in.
  - `d3d9_only_stubs.c`: no-op `run_shader_tests_d3d11`/`_d3d12` so the shared
    dispatch links without the other executors' import libraries.
- `corpus/*.shader_test`: the hand-assembled d3dbc-hex corpus (no HLSL compiler
  needed). Each file is a `[require]` shader model, one or more shader blocks
  and `[test]` blocks with `probe` expectations. The runner still compiles any
  HLSL variant blocks through the wine builtin `d3dcompiler_47` at device-run
  time.

## How it builds

`meson.build` is the local wrapper. It mirrors the `tests/vkd3d-proton` shape:

- `dx9-shader-oracle-corpus-lint` (`corpus_lint.py`) is wine-free and GPU-free,
  so it joins the required `host` suite and runs on every configuration. It
  parses every `.shader_test` the way `shader_runner` would and fails on a
  malformed corpus edit, catching a bad corpus in CI before it reaches a device.
- `dx9-shader-oracle-runner.exe` is a Windows/PE binary built as part of the
  normal cross build (so a PR cloner's CI produces it). It needs `widl` to
  generate the `IDxcCompiler3` header; if `widl` is absent the runner build is
  skipped with a warning and the corpus lint still guards the corpus.
- The nine per-corpus device tests are in a separate `shader-oracle` suite, not
  `host`. They need a live Metal device plus a wine exe_wrapper to launch the
  PE, so meson reports them as skipped on any configuration without an
  exe_wrapper.

Two DXMT-tree build details differ from upstream vkd3d and are handled in
`meson.build` rather than by editing the vendored sources:

- The dxmt tree defines `NOMINMAX` project-wide (to keep the windows.h
  `min()`/`max()` macros out of the C++ frontend). vkd3d's `shader_runner`
  relies on those macros, which `vkd3d_windows.h` emits under `#ifndef NOMINMAX`,
  so the oracle TUs are compiled with `-UNOMINMAX` to use vkd3d's own
  definitions verbatim.
- The vendored `.c` are kept byte-identical to upstream, which does not build
  with `-Wimplicit-fallthrough`; the flag is suppressed for these TUs rather
  than annotating the upstream switch.

## Running against dxmt on a device

The device suite is skipped unless meson has a PE exe_wrapper (wine) configured.
On a device runner with a dxmt `d3d9.dll` on `WINEDLLPATH`:

```
meson test -C build --suite shader-oracle --print-errorlogs
```

Or drive a single case directly, which is what the maintainers use standalone:

```
WINEDLLPATH=/path/to/dxmt/prefix/dll wine \
  build/tests/dx9/shader-oracle/dx9-shader-oracle-runner.exe \
  tests/dx9/shader-oracle/corpus/pow.shader_test
```

The exe reads one `.shader_test`, draws each `[test]` block and compares every
`probe` against the render target at the stated ulps tolerance; a mismatch is a
nonzero exit with the failing probe printed. Do not weaken a probe tolerance to
paper over a dxmt numeric difference; a probe failure is the oracle working.

## Deferred

- HLSL corpus. The corpus is d3dbc-hex only, so no HLSL compiler is required to
  run it. The runner keeps upstream's optional `d3dcompiler_47` path for HLSL
  variant blocks, but the vendored corpus does not exercise it.
- The d3d11/d3d12/gl/vulkan executors are intentionally not vendored; only the
  d3d9 backend is relevant to this frontend.
